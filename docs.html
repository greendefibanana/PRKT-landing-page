Overview

What PRKT is and why it exists

Foundational

PRKT is a local-first, policy-enforced Solana agent wallet runtime. Each install owns its own state, each agent can have its own wallet, and every autonomous action still passes through policy, simulation, and execution controls before it can hit the chain.

What PRKT adds

Hard execution guardrails, wallet custody options, local audit state, proof anchoring, and an operator control surface.

What it is not

Not unconstrained autonomy, not a single hosted key for all users, and not a present-tense zero-knowledge system.

Who needs it

Funds, treasuries, protocols, automation teams, and products that need to prove an agent stayed inside policy.

Current truth: PRKT is live on Solana devnet today with policy enforcement, session/proof anchoring, compressed-account preference, and an onchain verifier path for managed execution.
Architecture direction: PRKT is being positioned around four compounding infrastructure lanes: compressed Solana state, Data Anchor-grade audit sealing, zkSVM-grade verifiable policy execution, and Neon EVM reach under the same policy boundary.
Quickstart

Fastest path from install to first governed agent run

npm install -g @prktsol/prkt
prkt init
prkt config show
prkt doctor

npm install
npm run cli -- --help

  • prkt config show prints cluster, wallet key source, and live flags.
  • prkt doctor checks RPC connectivity, signer posture, and local writability.
Architecture

How the runtime is layered

5 layers
1. CLI and operator control plane

Commands for initialization, wallets, agents, policy management, monitoring, audit, config, and diagnostics.

2. Local registry and persistent state

Wallets, agents, activity, and the local platform key live in a per-user state directory.

3. Wallet and transaction services

Encryption, balances, ATA management, transaction build/simulate/send/confirm, RPC failover, and post-transaction verification.

4. Policy and sandbox layer

Spend caps, allowlists, deny-unknown behavior, suspicious-delta rejection, emergency lock, and simulation requirement.

5. Agent runtime and protocol execution

Persistent agents, strategy execution, live-first protocol flows, and fail-closed unsupported paths.

Design rule: no strategy bypasses policy or signing boundaries. Autonomy exists inside the runtime, not outside it.
Custody

Custody model, local state, encryption, and recovery

Security-critical

The default PRKT custody model is local-per-user. Each installation owns its own encrypted wallet registry. That means 100 installs imply 100 custody boundaries, not one shared key server.

Local-first mode

PRKT generates or reuses a per-user platform key, encrypts wallet secrets at rest, and stores state in the OS app-data directory unless overridden.

Remote signer mode

Set REMOTE_SIGNER_URL, REMOTE_SIGNER_BEARER_TOKEN, and REMOTE_SIGNER_PUBKEY together to move signing off-box.

Recovery path

Wallet and agent creation return a one-time recoveryKey, required for later secret export.

Default state locations

  • Windows: %APPDATA%\PRKT\
  • macOS: ~/Library/Application Support/PRKT/
  • Linux: ${XDG_DATA_HOME:-~/.local/share}/prkt/
  • Override: PRKT_CLI_HOME

Typical files

  • cli-wallets.json
  • cli-agents.json
  • cli-activity.json
  • platform-wallet-master.key

Policy

How PRKT says yes, no, and not until it is safe

Presets

  • observe-only
  • simulate-only
  • auto-devnet-safe
  • guarded-live
  • custom

What policy constrains

Approval mode, spend, tx count, session TTL, mints, destinations, program IDs, opaque instructions, and simulation requirements.

Hard blocks

Dangerous SPL delegation, close-account drains, owner or mint authority changes, and deny-unknown instruction behavior by default.

prkt policy presets
prkt policy show --agent alpha
prkt policy set-preset --agent alpha --preset guarded-live
prkt policy set-limits --agent alpha --max-sol-per-tx-lamports 100000000 --max-transactions-per-session 3
prkt policy clear-overrides --agent alpha
Important: guarded-live enables live-capable behavior, but it does not disable guardrails.
Wallet workflows

Managed wallet lifecycle and treasury-style operations

Command-driven 
prkt wallet create --name treasury
prkt wallet fund --name treasury --sol 2
prkt wallet balance --name treasury
prkt token mint-demo --authority treasury --decimals 6 --amount 1000
prkt wallet balance-spl --name treasury --mint <mint-address>
prkt wallet transfer-sol --from treasury --to <recipient-pubkey> --amount 0.1
prkt wallet transfer-spl --from treasury --to <recipient-owner-pubkey> --mint <mint-address> --amount 10

Create behavior

PRKT generates a wallet, encrypts the secret, stores it locally, returns a one-time recovery key, and keeps registry state in sync.

Funding behavior

If a devnet treasury is configured, PRKT funds from that treasury first. Otherwise it uses an airdrop-oriented devnet path.

Agent workflows

Persistent agents, wallet attachment, and runtime state

An agent in PRKT is a persistent identity with a managed wallet, policy preset, optional overrides, strategy choice, and runtime activity history.

Create

prkt agent create provisions the wallet, records the agent, attaches the default policy, and returns the recovery key.

Inspect

prkt agent show returns persisted state plus the resolved policy view.

Operate

run, run-all, logs, balance, and stop make agents usable in scheduler or operator-led loops.

Execution model

Strategies, live paths, simulation, and fail-closed behavior

Reality check

Current strategy names

  • memo-heartbeat
  • simple-scripted-transfer
  • universal-defi

Execution modes

PRKT supports live execution, simulation-only execution, and live-first with simulated fallback depending on cluster and protocol support.

Fail-closed rule

Unsupported protocol or EVM paths must not silently become ungoverned execution.

Neon expansion path

Neon extends PRKT's policy surface toward EVM execution, but live broadcast remains explicitly gated so unsupported paths fail closed instead of bypassing controls.

Accurate claim: some paths are live on devnet now, some are intentionally simulated, and some are live-first with fallback.
Proofs and verification

Session anchoring, policy attestation, compressed storage, and the onchain verifier

Live on devnet

PRKT's current proof model is cryptographically verifiable, but it is not zero-knowledge yet. There are two proof-verification paths live today.

The broader stack being sold to reviewers is bigger than today's minimum live path: Data Anchor-style tamper-proof session sealing, zkSVM-grade compliance verification, and a future zero-knowledge proof surface that lets third parties verify policy execution without trusting the operator.

Offchain attestation path

  • PolicyCircuit.prove() builds a signed policy attestation.
  • The payload includes intent hash, policy hash, and check outcomes.
  • ProofAnchor.anchorProof() stores the attestation with compressed-account preference.
  • ProofAnchor.verifyProof() reconstructs the digest and verifies the detached signature.

Onchain verifier path

  • programs/policy_guard is deployed on devnet.
  • The Solana Ed25519 precompile plus instruction introspection verify a signed managed-transfer payload onchain.
  • Current program ID: 3sUkfLW4jtwSQFgdtWyEj8FPedtvKfXSB1J16PMUZhMG.

Data Anchor direction

The audit model is aimed at a Data Anchor-grade outcome: session logs sealed into a public, tamper-evident commitment trail that outside reviewers can verify independently.

zkSVM direction

The long-form proof story is a zkSVM-style verifier path where policy execution moves from signed attestation toward stronger cryptographic compliance proofs.

Zero-knowledge upgrade

Today's live model is verifiable attestation plus an onchain managed verifier. The next upgrade is a true zero-knowledge proof path rather than a trust-me sandbox claim.

Session anchoring

SessionAnchor.startSession() and closeSession() write session commitments with compressed-account preference.

Fallback mode

In IMPLEMENTATION_PATH=defensible_devnet_demo, PRKT falls back to commitment-only anchors plus a local payload registry.

Audit surfaces

prkt audit --limit 200 and prkt agent logs --agent ... explain rejections and support incident review.

Emergency controls

Emergency lock can be set in the environment or file, agents can be stopped explicitly, and repeated failures can trigger circuit-breaker behavior.

Detect

Use audit trail, balance verification, tx history, and repeated failure signals.

Contain

Enable the emergency lock, pause the agent, and stop live execution before debugging the root cause.

Recover

Fix policy or config, clear the lock, start a fresh session, and verify balances after resuming.

Production guidance

How to describe and deploy PRKT honestly

Messaging-safe

Correct product statement

Each user gets local wallet custody by default, each install owns its own encrypted state, and agents act within policy constraints.

Claims to avoid

Do not call PRKT a shared wallet backend, unconstrained autonomy, or live zk proof infrastructure today.

Self-serve deployment

Use the default local state model, keep live flags intentional, and treat PRKT as a user-owned wallet runtime.

Managed operations

Use persistent PRKT_CLI_HOME, inject PRKT_WALLET_MASTER_KEY, and move signing behind a remote signer.

Reviewer messaging

Lead with policy enforcement, explorer-verifiable commitments, compressed-account preference, and explicit live-versus-roadmap boundaries.

Troubleshooting

Common failures and what they usually mean

Practical fixes

Wallet or agent not found

Check the current OS user and PRKT_CLI_HOME. PRKT state is local and per-install.

cli-data-dir-writable failed

Fix permissions or move the state with PRKT_CLI_HOME.

Session blocked: session has expired

Start a new session or increase POLICY_SESSION_TTL_MINUTES within the allowed range.

Program blocked or destination blocked

Check the preset, overrides, and allowlists attached to the agent.

Funding or RPC failures

Retry with a healthy endpoint, add SOLANA_RPC_FALLBACK_URL, or use a configured treasury wallet.

Live protocol path did not execute

Confirm the protocol live flag, cluster support, required config files, and actual devnet market health.

Debug habit: start with prkt config show, then prkt doctor, then agent show, policy show, and audit.
