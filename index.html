Deployment posture

Three ways to ship PRKT, one control philosophy.

PRKT can start as a local-first operator tool and still scale toward stronger signing and managed execution boundaries without throwing away the same policy model.

Local-first

For self-serve operators and rapid product onboarding.

Each install owns its own encrypted state, its own local key boundary, and its own agent wallets. That is the fastest path to "Phantom for agents" behavior.

Remote signer

For teams that need stronger production custody.

Swap local signing for `REMOTE_SIGNER_*` while keeping the same policy inspection, monitoring, and runtime controls in front of every execution attempt.

Managed verifier

For controlled vault release with onchain verification.

Use the deployed policy guard program for managed vault flows where the release of funds itself is subject to signed payload verification onchain.

Need-to-have agent wallet

If the agent can move money, the wallet has to be allowed to say no.

PRKT is how you make agent wallets defensible: tighter custody, stricter execution, better evidence, and a product story that stands up under scrutiny.
